# 数据契约

本项目所有解析、知识抽取、去重压缩、RAG 和教师编辑都围绕同一条证据链展开：

```text
RawFile -> DocumentElement -> Section -> Chunk -> KnowledgeNode / KnowledgeEdge -> Evidence / Decision / TeacherEdit
```

## 通用约束

- 每个对象必须有稳定 `id`。当前实现用来源文件 hash、页码、section 页范围、chunk 字符范围生成。
- `Chunk`、`KnowledgeNode`、`KnowledgeEdge`、`IntegrationDecision`、`IntegratedConcept`、`RetrievalEvidence` 必须直接携带 `source_locator` 或 `source_locators`。
- 自动合并、删除、压缩只能通过 `IntegrationDecision` 记录，不能覆盖原始证据。
- 教师修改通过 `TeacherEdit` 记录为事件，后续再触发局部再整合。

## SourceLocator

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `raw_file_id` | string | 来源文件 ID |
| `source_path` | string | 原始文件或转换目录路径 |
| `source_type` | string | `uploaded` / `converted_textbook` / `generated` |
| `locator_text` | string | 人类可读定位，例如 `生理学 pages 10-11` |
| `page_start/page_end` | number/null | PDF 或教材页码范围 |
| `line_start/line_end` | number/null | TXT/MD 行范围 |
| `sheet_name,row_start,row_end` | string/number/null | Excel 定位 |
| `slide_number` | number/null | PPT 定位 |
| `char_start/char_end` | number/null | 当前 section 内字符范围 |
| `element_ids` | string[] | 关联的 DocumentElement |
| `quote_hash` | string/null | 证据文本 hash，用于检测漂移 |

## RawFile

表示原始输入文件或已转换教材。

核心字段：`id`、`original_filename`、`title`、`format`、`source_type`、`storage_path`、`sha256`、`size_bytes`、`page_count`、`text_char_count`、`metadata`。

## DocumentElement

清洗后的最小文档元素。当前 converted_textbooks 导入器先按页生成 `page` element。

核心字段：`id`、`raw_file_id`、`type`、`text`、`order_index`、`parent_section_id`、`source_locator`、`char_count`、`metadata`。

可选类型：`heading`、`paragraph`、`table`、`figure`、`list`、`page`、`slide`、`sheet`、`note`。

## Section

面向解析和抽取的结构单元。当前第一阶段按 5 页窗口生成 `page_window` section，后续会升级为章节、slide、sheet 等真实结构。

核心字段：`id`、`raw_file_id`、`title`、`section_type`、`level`、`order_index`、`parent_section_id`、`element_ids`、`content`、`char_count`、`source_locator`、`metadata`。

## Chunk

RAG 和证据引用的基本单元。当前默认约 700 中文字符，80 字 overlap。

核心字段：`id`、`raw_file_id`、`section_id`、`text`、`order_index`、`char_start`、`char_end`、`char_count`、`source_locator`、`metadata`。

每个 chunk 的 `source_locator` 必须能回到页码、section 字符范围和关联 element。

## KnowledgeNode

知识点节点。

核心字段：`id`、`name`、`node_type`、`definition`、`aliases`、`source_locator`、`evidence_chunk_ids`、`confidence`、`metadata`。

节点类型：`Concept`、`Term`、`Mechanism`、`Process`、`Structure`、`Function`、`Disease`、`Symptom`、`Pathogen`、`Diagnosis`、`Treatment`、`Experiment`、`Example`。

## KnowledgeEdge

知识关系边。

核心字段：`id`、`source_node_id`、`target_node_id`、`relation_type`、`description`、`source_locator`、`evidence_chunk_ids`、`confidence`、`metadata`。

关系类型：`ALIAS_OF`、`SAME_AS`、`IS_A`、`PART_OF`、`PREREQUISITE_OF`、`CAUSES`、`LEADS_TO`、`APPLIES_TO`、`CONTRASTS_WITH`、`EXPLAINS`、`EVIDENCED_BY`、`MENTIONED_IN`、`REFINES`、`CONFLICTS_WITH`。

计划 03 新增关系类型：

- `CONTAINS`：章节或上位概念包含下位知识点。
- `PARALLEL_WITH`：两个知识点在同一章节中并列出现。

## GraphResponse

单本教材知识图谱输出。

核心字段：`id`、`raw_file_id`、`title`、`nodes`、`edges`、`generated_at`、`metadata`。

`metadata` 当前包含：

- `builder`：构建器名称。
- `prompt_version`：LLM prompt / 兜底抽取版本。
- `section_count`、`node_count`、`edge_count`。
- `llm_enabled`、`llm_cache_hits`、`llm_calls`、`llm_errors`。
- `fallback_sections`、`fallback_used`。

## GraphBuildRequest

触发单本教材图谱构建。

```json
{
  "raw_file_id": "raw_xxx",
  "section_ids": [],
  "force_rebuild": false,
  "max_sections": 20,
  "max_nodes_per_section": 8,
  "use_llm": true
}
```

## GraphNodeDetailResponse

节点详情响应，包含：

- `node`：`KnowledgeNode`。
- `edges`：与该节点相连的 `KnowledgeEdge`。
- `evidence_chunks`：节点证据 chunk。
- `graph_id`、`raw_file_id`。

## LayeredGraphResponse

阶段 6 新增的多层 KG 输出。它不是把 Tree-KG、Concept KG、Evidence Graph 混成一个黑箱，而是在同一个响应内保留 `layer_type`。

当前已生成真实内容的层：

- `document_tree`：`RawFile -> Section -> DocumentElement / Chunk`，含 `CONTAINS`、`HAS_ELEMENT`、`HAS_CHUNK`、`NEXT`。
- `concept_kg`：复用阶段 5 的 `KnowledgeNode / KnowledgeEdge`，表达知识点语义关系。
- `evidence_graph`：把知识节点、知识关系和 chunk 证据显式连接，关系包含 `EVIDENCED_BY`、`RELATION_SOURCE`、`RELATION_TARGET`。
- `evidence_graph` 还包含 `SourceQuote` 节点，用 `HAS_SOURCE_QUOTE` 和 `DERIVED_FROM` 把节点/边证据落回原文 chunk。

以下层只预留，不在阶段 6 生成真实整合结果：

- `alias_alignment`：阶段 7。
- `integration_decision`：阶段 8。
- `graphrag_retrieval`：阶段 9。
- `teacher_edit`：阶段 10。

核心字段：

- `layers`：`LayeredGraphLayer[]`，每层的展示名、状态、节点数、边数。
- `nodes`：`LayeredGraphNode[]`，每个节点都必须携带 `source_locator`。
- `edges`：`LayeredGraphEdge[]`，每条边都必须携带 `source_locator`；证据边还要携带 `evidence_chunk_ids`。
- Document Tree 顺序边包含 `NEXT` 和 `PREVIOUS`，便于后续教学路径和局部展开。

`LayeredGraphBuildRequest`：

```json
{
  "raw_file_id": "raw_xxx",
  "force_rebuild": false,
  "build_missing_concept_graph": true,
  "max_sections": 20,
  "max_nodes_per_section": 8,
  "use_llm": true
}
```

## RAG Evidence Index

阶段 4 先落地本地混合 Evidence Index：BM25 + lightweight hash embedding + query coverage。后续可替换为 FAISS / Chroma / 模型 embedding。索引对象仍然是 `Chunk`，引用仍然通过 `source_locator` 回到原文。

`RagIndexRequest`：

```json
{
  "raw_file_ids": ["raw_xxx"],
  "force_rebuild": false,
  "max_chunks": 10000
}
```

`RagIndexStatus`：

- `status`：`empty` / `ready`
- `textbook_count`
- `chunk_count`
- `raw_file_ids`
- `index_path`
- `updated_at`

`RagQueryRequest`：

```json
{
  "question": "动作电位是什么？",
  "top_k": 5,
  "raw_file_ids": []
}
```

`RagQueryResponse`：

- `answer`：带结构化引用的回答；没有命中时固定返回 `当前知识库中未找到相关信息。`
- `citations`：`RagCitation[]`
- `source_chunks`：原文 `Chunk[]`

`RagCitation` 必含：`chunk_id`、`raw_file_id`、`textbook`、`chapter`、`source_locator`、`relevance_score`、`quote`。

`RagCitation.metadata` 当前包含：

- `retrieval`：检索策略。
- `matched_terms`：命中的查询词。
- `query_coverage`：查询词覆盖率。
- `bm25_score`、`vector_score`、`raw_score`：检索分数组成。
- `quote_hash`：引用文本 hash。

## ConceptCluster

跨教材术语对齐和去重结果。

核心字段：`id`、`canonical_name`、`aliases`、`member_node_ids`、`evidence_chunk_ids`、`confidence`、`metadata`。

## AlignmentResponse

阶段 7 的术语对齐输出。它只回答“哪些节点可能是同一概念或别名”，不执行 `merge/keep/remove`，也不计算压缩比。

核心对象：

- `AlignmentCandidate`：两个节点之间的候选关系，关系类型为 `SAME_AS`、`ALIAS_OF`、`REFINES`、`CONFLICTS_WITH`。
- `AlignmentSignal`：候选关系的信号明细，包含 `normalized_name`、`alias_table`、`definition_similarity`、`context_similarity`、`neighbor_similarity`、`hash_embedding`。
- `ConceptCluster`：候选聚类，包含 `canonical_name`、`aliases`、`member_node_ids`、`confidence` 和证据 chunk。
- `CanonicalConcept`：聚类后的规范概念视图，保留所有 member node 和 source locator。
- `AliasRecord`：alias 到 canonical name 的映射记录。

`AlignmentBuildRequest`：

```json
{
  "raw_file_ids": ["raw_a", "raw_b"],
  "force_rebuild": false,
  "min_confidence": 0.62,
  "include_singletons": false,
  "max_nodes": 1000
}
```

阶段 7 当前使用本地多信号评分：

- 名称规范化。
- 内置 alias/canonical 表。
- 定义 token 相似度。
- 章节上下文相似度。
- 图谱邻居相似度。
- lightweight hash embedding。

低置信候选和 `CONFLICTS_WITH` 会标记 `needs_teacher_review=true`，等待阶段 8/10 的决策或教师复核。

## IntegrationDecision

压缩和整合决策。

核心字段：`id`、`cluster_id`、`action`、`target_node_ids`、`retained_content`、`removed_redundancy`、`reason`、`confidence`、`evidence_chunk_ids`、`source_locators`、`created_at`、`metadata`。

动作类型：`merge`、`keep`、`remove`、`refine`、`conflict`。

阶段 8 约束：

- `merge` 只表示跨教材同一/别名概念合并，不删除原始节点。
- `remove` 只表示从整合正文移出低价值冗余内容，原始 KG 和证据仍保留。
- `refine` 用于粒度更细或补充说法，不直接压平为同一文本。
- `conflict` 用于矛盾说法，强制保留双方证据并等待教师裁决。
- 每条决策必须有 `reason`、`confidence`、`evidence_chunk_ids` 和 `source_locators`。

## IntegratedConcept

阶段 8 的整合后概念视图，不替代原始 `KnowledgeNode`。

核心字段：`id`、`canonical_name`、`member_node_ids`、`decision_ids`、`definition`、`summary`、`source_locators`、`evidence_chunk_ids`、`confidence`、`metadata`。

## CompressionStats

阶段 8 的压缩统计。

核心字段：`original_char_count`、`retained_char_count`、`original_node_count`、`integrated_node_count`、`merged_node_count`、`kept_node_count`、`removed_node_count`、`refined_node_count`、`conflict_count`、`target_compression_ratio`、`compression_ratio`、`node_reduction_ratio`、`evidence_coverage_ratio`、`metadata`。

`original_char_count` 优先按解析后的 `Section.char_count` 统计；只有缺失解析结果时才回退到图谱节点定义长度。`compression_ratio = retained_char_count / original_char_count`。

## IntegrationResponse

阶段 8 的跨教材整合与压缩输出。

核心字段：

- `id`
- `raw_file_ids`
- `alignment_id`
- `decisions: IntegrationDecision[]`
- `integrated_concepts: IntegratedConcept[]`
- `compression_stats: CompressionStats`
- `generated_at`
- `metadata`

`IntegrationBuildRequest`：

```json
{
  "raw_file_ids": ["raw_a", "raw_b"],
  "force_rebuild": false,
  "target_compression_ratio": 0.3,
  "alignment_min_confidence": 0.62,
  "include_keep_decisions": true,
  "max_nodes": 1000
}
```

## TeacherEdit

教师修改事件。

核心字段：`id`、`target_type`、`target_id`、`operation`、`before`、`after`、`reason`、`created_by`、`created_at`、`affected_ids`、`metadata`。

操作类型：`create_node`、`update_node`、`delete_node`、`split_node`、`merge_nodes`、`create_edge`、`update_edge`、`override_decision`。

## RetrievalEvidence

RAG / GraphRAG 检索证据。

核心字段：`id`、`query`、`chunk_id`、`raw_file_id`、`source_locator`、`relevance_score`、`answer_span`、`metadata`。

## ApiErrorResponse

所有 HTTP 错误统一返回：

```json
{
  "message": "教材解析失败",
  "code": "PARSE_FAILED",
  "detail": "原始错误信息"
}
```

字段约定：

- `message`：给 UI 展示的人类可读错误。
- `code`：稳定错误码，例如 `PARSE_FAILED`、`VALIDATION_ERROR`、`HTTP_404`。
- `detail`：调试信息，可为空；前端不应依赖其结构。

## JobRecord 与 PipelineStepRecord

`JobRecord` 是阶段 3 异步任务 DAG 的状态载体。除基础字段外，新增：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `steps` | `PipelineStepRecord[]` | 当前任务的步骤列表 |
| `retryable` | boolean | 失败后是否可重试 |
| `context_path` | string/null | 后端内部重试上下文路径，前端只读展示即可 |

`PipelineStepRecord`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `name` | enum | `assemble_file`、`detect_format`、`parse_elements`、`build_sections`、`chunk_sections`、`persist_parsed` |
| `status` | enum | `queued`、`running`、`completed`、`failed`、`skipped` |
| `progress` | number | 单步骤进度 |
| `message` | string | 当前步骤说明 |
| `started_at/finished_at` | datetime/null | 步骤时间 |
| `error` | string/null | 步骤错误 |

## 计划 01/02 导入输出

调用：

```text

```

若不上传文件，接口会从 `materials/converted_textbooks` 导入第一本教材，生成：

```text
data/parsed/{raw_file_id}.json
```

JSON 顶层结构：

```json
{
  "id": "parsed_xxx",
  "raw_file": {},
  "elements": [],
  "sections": [],
  "chunks": [],
  "generated_at": "2026-05-10T...",
  "metadata": {
    "source_pipeline": "materials.converted_textbooks",
    "chunk_size": 700,
    "chunk_overlap": 80

```

当前 converted_textbooks 导入器已加入最小结构清洗：

- 跳过封面、版权、编委、前言、目录等正文前噪声页。

- 删除重复页眉、纯页码、数字资源提示等展示噪声。
- 保留原始页级 `DocumentElement`，清洗后的 `Section` 和 `Chunk` 仍通过 `source_locator.page_start/page_end` 追溯回原文页。

## 多格式解析定位约定

| 格式 | section 策略 | source_locator |
| --- | --- | --- |
| `txt` | 标题正则或固定长度窗口 | `line_start/line_end` |
| `md` | Markdown `#` 标题 | `line_start/line_end` |
| `pdf` | 章节正则优先；失败时 5 页窗口 | `page_start/page_end` |
| `docx` | Word Heading 分 section，表格转 Markdown 并保留原位置 | `line_start/line_end` 表示段落或表格块范围 |
| `xlsx` | 每个 sheet 按 40 行窗口切分 | `sheet_name` + `row_start/row_end` |
| `csv` / `tsv` | 自动识别分隔符或按 TSV 读取，按 40 行窗口切分 | `sheet_name=CSV` + `row_start/row_end` |
| `pptx` | 每页 slide 一个 section，抽取标题、正文、表格、备注 | `slide_number` |

旧版 `doc/xls/ppt` 暂不直接解析，后续通过 LibreOffice 转换为 `docx/xlsx/pptx` 后再进入统一解析层。

## 批量上传响应

`POST /api/textbooks/upload-batch` 返回：

```json
{
  "job": {},
  "items": [],
  "errors": [],
  "total_count": 2,
  "success_count": 1,
  "failed_count": 1
}
```

- `items` 中每个元素与单文件上传响应一致，包含 `parsed_textbook`。
- `errors` 中记录失败文件名、错误信息和对应子任务 `job`。
- 只要至少一个文件成功，批量任务状态为 `completed`；全部失败时批量任务状态为 `failed`。

## 大文件分片上传

创建会话：

```json
{
  "filename": "large_textbook.pdf",
  "total_size_bytes": 104857600,
  "total_chunks": 100,
  "chunk_size_bytes": 1048576,
  "sha256": "完整文件 sha256，可选但推荐",
  "content_type": "application/pdf",
  "parse_on_complete": true
}
```

上传分片：

```text
PUT /api/uploads/sessions/{session_id}/chunks/{chunk_index}
```

`chunk_index` 从 0 开始，可以乱序上传。非最后一个分片大小必须等于 `chunk_size_bytes`。会话状态会返回：

- `uploaded_chunks` / `missing_chunks`
- `received_bytes`
- `upload_progress`
- `parse_progress`
- `assembled_path`
- `raw_file_id`
- `parsed_output_path`

完成会话：

```text
POST /api/uploads/sessions/{session_id}/complete
```

后端会按序合并分片，校验 `sha256`，然后在 `parse_on_complete=true` 时复用现有多格式解析器生成统一 JSON。PDF/Office 仍需要完整文件后解析；TXT/MD/CSV/TSV 当前也先走最终统一解析，后续可在该会话模型上扩展真正流式预解析。

异步版本：

```text
POST /api/uploads/sessions/{session_id}/complete-async
```

该接口立即返回 `JobRecord`，后台执行：

```text
assemble_file -> detect_format -> parse_elements -> build_sections -> chunk_sections -> persist_parsed
```

失败时 `job.retryable=true`，可调用：

```text
POST /api/jobs/{job_id}/retry
```

## OpenAPI 快照

后端契约事实来源是 `backend/app/models/schemas.py` 和运行时 `/openapi.json`。
每次新增 endpoint、schema 或错误响应结构后，后端需要导出快照：

```powershell
cd D:\Hackathon
$env:PYTHONPATH=(Resolve-Path backend).Path
backend\.venv\Scripts\python.exe backend\scripts\export_openapi.py
```

输出文件：`docs/openapi.snapshot.json`。

前端离线 codegen 应优先读取该快照，避免依赖后端 live 端口。当前 live 后端默认端口是 `8010`：

```text
http://127.0.0.1:8010/openapi.json
```
