# 阶段 8 整合压缩审计

## 阶段边界

阶段 8 只负责把阶段 7 的术语对齐候选转成可审计的整合和压缩决策：

- 去重判断：`merge`、`refine`、`conflict`。
- 压缩判断：`keep`、`remove` 和压缩后 `IntegratedConcept.definition`。
- 原始 `KnowledgeNode`、`KnowledgeEdge`、`Chunk` 不被覆盖或删除。

## 当前实现

- API：`POST /api/integration/build`、`GET /api/integration`。
- 输出：`IntegrationResponse`。
- 决策：`IntegrationDecision`，动作包括 `merge`、`keep`、`remove`、`refine`、`conflict`。
- 统计：`CompressionStats` 记录原始字符数、保留字符数、目标压缩率、实际压缩率、节点变化和证据完整度。
- 概念视图：`IntegratedConcept` 保留 `member_node_ids`、`decision_ids`、`source_locators`、`evidence_chunk_ids`。

## 问题清单对齐

| 问题 | 当前回答 | 验证方式 |
| --- | --- | --- |
| 是否区分去重和压缩？ | 是。`merge/refine/conflict` 先判断概念关系，`keep/remove` 再处理正文保留。 | 检查 `IntegrationDecision.action` 分布 |
| 是否能解释每个决策？ | 是。每条决策有 `reason`、`confidence`、`metadata`。 | smoke/benchmark 要求证据完整度 1.0 |
| 是否能追溯证据？ | 是。每条决策和整合概念都带 `evidence_chunk_ids`、`source_locators`。 | smoke 遍历所有 decision/concept |
| 是否避免直接删除矛盾？ | 是。矛盾输出 `conflict`，并强制保留双方节点。 | benchmark 要求 `conflict_count > 0` |
| 是否处理“进一步说法”？ | 是。定义粒度差异输出 `refine`，保留主定义和补充差异。 | benchmark 要求 `refine` 覆盖 |
| 是否控制压缩率？ | 是。按解析后的 `Section.char_count` 计算预算，低重要度 singleton 先移出整合正文，再按预算截断。 | benchmark 要求 `compression_ratio <= 0.30` |
| 是否保留教学链路关键点？ | 部分完成。当前用频次、图谱度数、证据数量和核心词加权，下一阶段可接教师规则。 | `metadata.frequency/degree` 和 keep/remove reason |

## 剩余风险

- 当前压缩重要度是本地规则，不等于真实教师偏好。
- `conflict` 只做检测和保留，不做裁决。
- `remove` 只移出整合正文，不代表从知识库删除；前端展示需要明确区分。
- 后续阶段 10 需要允许教师覆盖决策并触发局部重算。
