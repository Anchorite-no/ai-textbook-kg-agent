# 需求分析

## 目标

本项目面向教师的多教材知识整合场景，将 7 本医学教材解析为统一证据链，再构建知识图谱、跨教材对齐、压缩决策和 GraphRAG 问答能力。核心目标不是简单摘要，而是让每个知识点、关系、整合决策和回答都能追溯到原始教材位置。

赛题目标可拆成三条主线：

- 多格式教材进入统一数据契约：`RawFile -> DocumentElement -> Section -> Chunk`。
- 多层知识图谱与跨教材整合：单书 KG、证据图、术语对齐、整合决策。
- 面向教学使用的可解释问答：RAG / GraphRAG 引用原文，能解释来源、差异、前置知识和合并原因。

## 用户与场景

主要用户是课程教师和助教。典型任务包括：

- 上传或导入教材，查看章节结构、页码、chunk 和解析质量。
- 查看一本教材的知识点、关系和来源证据。
- 比较多本教材中的重复、互补、冲突和粒度差异。
- 将多本教材压缩为不超过 30% 的教学精华，同时保留证据链。
- 用自然语言追问某个概念是什么、出现在哪些教材、为什么合并或保留。

## 数据粒度

后端数据分为四级：

- `RawFile`：文件级稳定 ID，记录文件来源、格式、sha、页数和字符数。
- `DocumentElement`：页、段落、表格、幻灯片等原始元素，保留 source locator。
- `Section`：章节或小节，是教材结构展示和 KG 抽取的主要单位。
- `Chunk`：RAG 检索单元，当前约 700 字符、80 字符 overlap，metadata 保留章节和定位信息。

所有后续对象必须保留 `source_locator` 或 `evidence_chunk_ids`，以便追溯到页码、行号、sheet 行、slide 或原文片段。

## 重复判定

跨教材重复不是只看字符串相同，而是分层判断：

- 规范化名称相同或互为别名时，优先进入 `SAME_AS` 候选。
- 名称不同但上下文、定义、邻接关系相似时，进入语义对齐候选。
- 定义粒度不同但指向相近概念时，输出 `REFINES`，避免误合并。
- 定义或语义方向存在冲突时，输出 `CONFLICTS_WITH` 并保留双方证据。

当前实现使用本地规则、别名表、上下文相似度、邻接相似度和 hash embedding 组合打分；LLM KG 抽取已接入 Responses API，可进一步提升概念和关系质量。

## 压缩比要求

赛题要求多教材压缩到 30% 以内。当前后端将压缩分成两个层面：

- 概念层去重：`merge/refine/conflict` 处理跨书概念关系。
- 正文层压缩：按目标字符预算保留高价值定义，低价值 singleton 可移出整合正文。

7 本书全流程试跑中，解析后 section 字符数约 3,023,632，整合保留 225,283，压缩率 7.45%，低于 30% 目标；证据覆盖率为 100%。

## RAG 分块依据

当前 chunk size 为 700 字符，overlap 为 80 字符，理由是：

- 医学教材定义和机制解释通常跨 1-3 个自然句，700 字符能容纳完整上下文。
- 80 字符 overlap 能降低章节切分处丢失定义、因果和前置关系的概率。
- chunk metadata 保留 `section_id`、章节标题和 source locator，便于回答时引用教材、章节、页码或行号。

## 验收重点

后端验收优先级如下：

- 能导入 7 本教材并形成统一 JSON。
- 每个 chunk、node、edge、decision 都能追溯证据。
- 单书 KG 至少覆盖 `prerequisite/parallel/contains/applies_to` 等关系，并能扩展到 causes、part_of、explains、contrasts_with。
- 跨教材整合能输出可解释的 merge、keep、remove、refine、conflict 决策。
- GraphRAG 能返回 citations、source chunks、node hits、paths 和 integration decisions。
